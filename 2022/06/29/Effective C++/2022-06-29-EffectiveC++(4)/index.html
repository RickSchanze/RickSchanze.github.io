EffectiveC++(4)(18~25)

发表于2022-06-29|更新于2022-06-29|Effective C++
|阅读量:

18：让接口容易被正确使用，不易被误用

假如为一个表现日期的类设计构造函数：

class Date
{
    public:
    Date(int month, int day, int year);
};

在调用时可能会调用

auto date = Date(2, 30, 2022);

为防止这样的错误，可以导入新类型（利用类型系统）。

现在为上述例子导入简单的外覆类型来区别天数，月份与年数。

struct Day
{
    explicit Day(int d): val(d) {}
    int val;
};

struct Month
{
    explicit Month(int month): val(month) {}
    int val;
};

struct Year
{
    explicit Year(int year): val(year) {}
    int val;
};

class Date
{
    public:
    Date(const Month& month, const Day& day, const Year& year);
};

此时再进行调用时，可以保证错误的传入不过编译。

auto d1 = Date(2, 30, 1998);                        // wrong
auto d2 = Date(Day(30), Month(2), Year(2022));      // wrong
auto d2 = Date(Month(3), Day(2), Year(2022));       // right

可见，明智而审慎地导入新类型对预防“接口被误用”有奇效。

当一个类型地值有所限制时，我们可以预先定义其值，例如：

class Month
{
private:
    int val;

public:
    explicit Month(int month) : val(month) {}
    static Month Jan() { return Month(1);}
    static Month Fed() { return Month(2);}
    //...
};

预防错误地另一个方法是限制类型内什么事可以做，什么事不可以做，常见地限制为加上const，可回忆条款三。

另一个一般性准则为"除非有好理由，否则应让自定义类型行为与内置类型行为一致"。

例如对于int类型的a与b，对a*b赋值并不合法，那么自定义类型的"*"操作也不应能赋值。

避免无端与内置内省不兼容，其真正的理由是为了提供行为一致的接口。

任何接口如果要求客户必须记得做某些事情，则存在着”不正确使用“的倾向，以为客户有可能忘记。

例如一个条款13的工厂函数：

Investment* create();

此函数返回的指针应被包覆于智能指针中，但考虑到客户有可能忘记，将其改为这样的形式往往是更好的选择：

shared_ptr<Investment> create();

这样几乎不存在忘记的可能。

假如类设计者希望可以从create返回的指针需要由函数getRidOfInvestment来销毁，而不是直接delete，那么可以为其提供一个删除器：

shared_ptr<Investment> create()
{
    shared_ptr<Investment> retVal(nullptr, getRidOfInvestment);
    retVal = ...//正确的指向
    return retVal;
}

shared_ptr另一个特别好的性质是：它会自动使用它的“每个指针专属的删除器”，这样可以避免另一个错误：“跨DLL内存分配”。

但是shared_ptr比原始指针大且慢，且使用辅助动态内存，但它“降低客户错误”的成效却是有目共睹的。

请记住：

  • 好的接口很容易被正确使用，不容易被误用，应在自定义接口中努力达成这一性质
  • “促进正确使用”的办法包括接口的一致性，以及于内置类型行为兼容
  • “阻止误用”的办法包括建立新类型、限制类型上的操作，束缚对象值以及消除客户的资源管理责任
  • shared_ptr支持自定义删除器，这可以防范DLL问题，可被用来自动解除互斥锁等等

19：设计class犹如type

好的type拥有自然的语法，直观的语义以及一个或多个高效实现品，因此再设计class时，应该确认：

  • 新type的对象应该如何被创建和销毁？
  • 对象的初始化和对象的赋值该有什么样的差别？
  • 新type的对象如果以值传递，意味着什么？
  • 什么是新type的“合法值”？
  • 你的新type需要配合某个继承体系吗（主要考虑虚函数的问题）？
  • 你的新type需要什么样的转换？
  • 什么样的操作符和函数对此新type而言是合理的？
  • 什么样的标准函数应该被驳回？
  • 谁该群用新type的成员？
  • 什么是新type的“未声明接口”？
  • 你的新type有多么一般化？
  • 你真的需要一个新type吗？

20：宁以传常量引用代替传值

缺省条件下C++以值传递方式传递参数,而这造成的拷贝复制运算可能会造成巨大的开销,例如

bool validateStudent(Student s);

为了传入s(一个student对象),可能会在构造时多次拷贝复制(例如string),析构时又额外进行析构.

而使用const Student& s可避免这一问题

bool validateStudent(const Student& s);

const保证s不会被改变.

同时,传引用还可以避免对象切割(sliding)问题.当一个子类对象以传值的方式并且被视为一个基类对象时,基类的拷贝构造函数会被调用,而属于子类的那些特性会被全部切割,仅留下基类对象的特性,这往往与我们的想法背道而驰.

请记住：

  • 尽量以传递常量引用的方式代替值传递,前者较为高校,且可以避免对象切割的问题
  • 上述规则并不适用于内置类型,以及STL的迭代器和函数对象.对它们而言,传值较为妥当.

21：必须返回对象时，请不要返回其引用

下来我我们来看几个反例：

class Test {
public:
    friend const Test& operator*(const Test& lhs, const Test& rhs);
}

  • 假如使用这种方式

    const Test& operator*(const Test& lhs, const Test& rhs) {
    Test newTest;
    // do something
    return newTest;
}

    那么由于时在栈上分配的，最后返回的将是一个指向实际并不存在的对象的引用从而造成UB

  • 假如使用这种方式

    const Test& operator*(const Test& lhs, const Test& rhs) {
    Test newTest = new Test;
    // do something
    return *newTest;
}

    这看起来没问题，但是如果用户这样调用：

    Test a, b, c;
auto d = a *b *c;

    这会被展开为auto d = operator*(operator*(a, b), c)，其中a与b相乘所分配的内存将无法回收而造成内存泄漏。

  • 假如使用这种方式：

    const Test& operator*(const Test& lhs, const Test& rhs) {
    static Test newTest;
    // do something
    return Test;
}

    那么假如用户这样调用：

    Test a, b, c, d;
if ((a * b) == (c *d)) {
    // ...
}

    那么该if条件将永远为真，失去原本意图。

  • 使用static数组有多坏也不用说了。

那么我们可以怎么做呢？那就是不用引用：

class Test {
public:
    friend const Test operator*(const Test& lhs, const Test& rhs);
}

即可避免问题。

请记住：

  • 绝不要返回指向一个本地栈对象的指针或引用，或者返回指向堆对象的引用，或者是在需要多个对象的情况下返回本地静态对象

22：将成员声明为private

声明为private的理由：

  • 语法一致性:当客户拿到一个类时，若类里只有成员函数那么就不用记该不该加()了
  • 自由控制访问权限
  • 封装:若将成员声明为public,那么当改变该成员时（删去或者其他），使用者便需要进行大量改动，而如果声明为protected,其继承类也需要进行大量改动，严重拖慢开发效率
  • 可以使得成员变量被读写是轻松通知其他对象，可以验证class的约束条件以及函数的前提和事后状态，可以在多线程环境中执行同步控制…

请记住：

  • 切记将成员遍历声明为private,这可以赋予客户访问数据的唯一性，可细微划分访问控制，允诺约束条件获得保证，并提供class作者以充分的实现弹性
  • protected并不比public更具有封装性

23：宁以non-member,non-friend替换member函数

考虑一个类:

class WebBrowser{
    public:
    ...
    void clearCache();
    void clearHistory();
    void removeCookies();
    ...
}

因为可能会想一次执行所有，那么便有了两种选择

// 1. member
class WebBrowser{
    ...
    void clearEverything(); //调用上面写的三个函数
}
// 2. non-member, non-friend
void clearBrowser(WebBrowser& wb); // 调用

从面向对象的角度来看，member函数比non-member, non-friend函数带来的封装性低，因为member函数可以自由地访问私有变量而non-member, non-friend函数不能。越多的函数可以访问一个数据，那么该数据的封装性越低。

因此封装性较高的是non-member, non-friend函数，因为它并不增加“能够访问”class内部private成员的函数数量。

但是有两件事值得注意：

  • 这个结论仅适用于non-member, non-friend函数和member函数之间
  • 函数已经是一个类的non-member, non-friend函数了，但这并不意味这它不能成为另一个类的member函数

在C++中比较自然的做法是声明于namespace中：

namespace WebBrowserStuff{
    class WebBrowser{...};
    void clearBrowser(WebBrowser& wb);
}

由于namespace可以跨文件扩展，因此使用namespace又可以达到“按需include”的效果，只需将便利函数置于不同头文件的同一命名空间即可。

同时将所有便利（工具）函数放在多个头文件可以减少文件之间的耦合性，使代码更清晰。客户也可以轻松对其拓展。

请记住：

  • 宁可使用non-member, non-friend函数代替member函数，这么做可以增加封装性，降低耦合性和机能扩充性。

24：若所有参数均需要类型转换，请问此使用non-member函数

假如有一个数值类型：

class Rational
{
public:
	Rational(int number, int denominator = 1);		// 允许隐式转换
    int numerator() const;
    int denominator() const;
}

现在又想支持一个乘（或加）法操作，如果你将其写入成员函数：

class Rational
{
public:
	const Rational operator*(const Rational&);
}

则

Rational a(3, 2);
auto b = a * 2;	// 通过编译
auto c = 2 * a;	// 不通过编译

违反了交换律，很不舒服，因此可以改成

class Rational{};
const Rational operator*(const Rational& lhs, const Rational& rhs);
Rational a(3, 2);
auto b = a * 2;	// 通过编译
auto c = 2 * a;	// 通过编译

这样便同时支持了隐式转换和交换律。

同时还应注意：无论何时如果可以避免友元函数就应该予以避免

请记住：

  • 如果你需要为某个函数的所有参数（包括被this指针所指的那个隐式参数）进行类型转换，那么这个函数必须是个非友元函数。

25：考虑写出一个不抛异常的swap函数

标准库的swap函数长这样：

namespace std{
    template <typename T>
    void swap(T& a, T& b) {
        T temp(a);
        a = b;
        b = temp;
    }
}

这是基于拷贝赋值运算符的，因此可能造成效率问题。

假如你的类十分大，拷贝赋值会造成大量性能损失，那么可以使用所谓的pimpl手法（pointer to implementation）来解决问题。

假设：

class WidgetImpl
{
public:
private:
    int a, b, c;
    std::vector<double> v; //可能有很多数据，这意味复制时间很长
};
class Widget
{
public:
    Widget(const Widget &rhs);
    Widget &operator=(const Widget &rhs)
    {
        //...
        *pImpl = *(rhs.pImpl);
        //...
    }

private:
    WidgetImpl *pImpl;
};

在调用swap函数时其实只需要置换指针，但是swap不知道，它仍然调用了三次赋值函数。

可以这样：

namespace std
{
    template <>
    void swap<Widget>(Widget &a, Widget &b)
    {
        swap(a.pImpl, b.pImpl);
    }
}

这是一模板的全特化，但由于pIml为私有成员，所以无法通过编译

因此我们可以这样做：

class Widget
{
public:
    void swap(Widget &other)
    {
        using std::swap;
        /*令swap可用
          根据C++名称查找规则，会先查找合适的对应的上的
          之后考虑可能以及特化了的std::swap
        */
        swap(pImpl, other.pImpl);
    }
}
namespace std
{
    template <>
    void swap<Widget>(Widget &a, Widget &b)
    {
        a.swap(b);
    }
}

即令swap调用成员函数。

假如是类模板，则需要在其他命名空间里声明而不能在std中声明，因为std不能加新东西，但可以对模板全特化

请记住：

  • 当std::swap对你的类型效率不高时，提供一个swap成员函数，并保证它不抛出异常(因为swap的一个应用是提供强烈的异常安全性)
  • 如果提供了一个member swap,也应该提供一个non-member来调用前者，对于class,请特化std::swap
  • 调用swap时应针对std::swap使用using形式，然后调用swap并且不到任何命名空间修饰
  • 为用户自定义类型进行std template进行全特化是好的，但千万不要尝试在std内引入对std而言全新的东西
